# Spec 2.2 — Committed example, dashboard screenshot, README value overhaul

**Review findings:** DESIGN_REVIEW.md D1 (High), P2, P4, D2 (Medium). **Effort:** S–M. **Depends on:** spec 2.1 (dogfood) recommended first; specs 1.1/1.2 recommended first so the screenshot shows coverage + stale sections.

## Goal

A first-time visitor sees the product's output within ten seconds of opening the README, can run the report against committed example data without installing anything into their own repo, and gets answers to the two silent adoption blockers: "what does this cost?" and "how do I know it's working?".

## Requirements

### R1 — `examples/demo`: a committed end-state repo

Create `examples/demo/` containing a small, realistic migrated project (invent a plausible one, e.g. a Node web app; or derive from `evals/fixtures/single-claude-md` — but as *output*, i.e. already migrated):

- `.agents/rules/root.md`, `.agents/rules/testing.md` (~6–10 rules total, standard anatomy, mixed severities including at least one MUST).
- `.agents/rules-catalog.md` generated by `generate-catalog.mjs --write`.
- `.agents/rule-trace.md` from the template.
- `CLAUDE.md` + `AGENTS.md` thin importers; `.agents/rule-trace.config.json` scoping importers to those two.
- `.agents/metrics/traces.jsonl` — **committed** seeded data: ~30–60 hand-crafted events with realistic UUIDs and timestamps spread over ~6 weeks, engineered so the dashboard shows every flag class: at least one dead rule, one always-candidate-never-applied, one low-rate, one un-waived MUST gap, one unknown ID, one stale rule, and (if spec 1.1 landed) a coverage around 60–80% via some `traced:false` events.
- The generated `.agents/metrics/report.json` and `dashboard.html`, committed.

Gitignore check: the root `.gitignore` patterns for metrics files contain slashes and are therefore anchored to the repo root — they must **not** ignore `examples/demo/.agents/metrics/*`. Verify with `git check-ignore -v`; adjust patterns if needed (keep root metrics ignored).

Add `examples/demo/README.md` (10 lines): what this is, and the two commands to regenerate:

```bash
node skills/rule-trace/scripts/validate-rules.mjs --root examples/demo
node skills/rule-trace/scripts/report.mjs --root examples/demo
```

### R2 — Dashboard screenshot

Render `examples/demo/.agents/metrics/dashboard.html` in a browser (Playwright + the preinstalled Chromium; viewport ~1100×900, light scheme) and save a PNG to `docs/dashboard.png`. Crop/size so the stat tiles, the warning note, and the top of the rule table are visible. Embed it near the top of the README (after the intro paragraph, before "Why Use It") with alt text.

### R3 — README value framing (two tiers)

Rework the README's opening (`README.md:1-42`) to lead with the immediate payoff, then the long-run payoff:

- **Tier 1 (today):** every substantive agent response carries a clickable, reviewable trace — reviewers see *which* rules shaped the work and *why* one was waived, instead of trusting "the agent followed the rules."
- **Tier 2 (over weeks):** the counters tell you which rules to delete, narrow, or rewrite.

Keep the existing honest-limits content intact. Do not grow the README's total length materially — tighten elsewhere to compensate.

### R4 — Cost subsection

Add a short "What it costs" subsection (3–5 sentences, under "Limits" or as its own section): a trace block is roughly 50–100 tokens per substantive response; rule files replace prose the entry points were already loading, so migration is roughly context-neutral; the convention file adds a few hundred tokens once per session; the Stop hook is a local subprocess per turn that never blocks the agent. Present these as rough orders of magnitude, not benchmarks.

### R5 — "Verify it's working" quickstart step

Add to the Quickstart (`README.md:142-157`): after install + migrate, ask the agent one rule-shaped question (give a copy-paste example prompt), confirm the response ends with a `Rule trace` block, and — if the live hook is wired — confirm `.agents/metrics/traces.jsonl` gained a line. Three or four lines total. Link the demo: "or try the report on the committed example first: `node <skill>/scripts/report.mjs --root examples/demo`".

## Acceptance criteria

1. `validate-rules.mjs --root examples/demo` exits 0, no errors, no warnings.
2. `report.mjs --root examples/demo --out-json /tmp/r.json` succeeds and the flags include ≥1 entry in **each** of: dead, always-candidate-never-applied, lowRate, unwaivedMustGaps, unknownIds, stale (if 1.2 landed).
3. Committed `report.json`/`dashboard.html` byte-match a fresh regeneration (run and diff), except the `generatedAt` timestamp line — note this exception in the demo README.
4. `docs/dashboard.png` exists, renders the seeded data (not an empty state), and the README displays it.
5. README contains the two-tier framing, the cost subsection, and the verify step; total README line count grew by less than ~40 lines net.
6. `git check-ignore` confirms no demo file is ignored; `npm pack --dry-run` confirms `examples/` and `docs/` are **not** in the published tarball (the `files` allowlist already excludes them — verify).
7. `npm test` green.
